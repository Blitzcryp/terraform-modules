# Module Conventions & Standards

> The contract every module in this repo must follow. Atoms, components, and
> blueprints all obey these rules. When in doubt, copy `atoms/kms-key` (the
> reference implementation) and `_template/atom`.

## 1. The three layers

```
atoms        →  components      →  blueprints
(1 resource)    (1 capability)     (1 deployable environment)
```

| Layer         | Owns                          | Creates dependencies? | Example |
|---------------|-------------------------------|-----------------------|---------|
| **Atom**      | Exactly one logical resource  | **No** — takes IDs/ARNs as inputs | `security-group`, `s3-bucket`, `kms-key` |
| **Component** | One capability = several atoms | Only atoms it fully owns | `private-encrypted-bucket`, `bastion-host` |
| **Blueprint** | One environment = several components | Only components | `pci-3tier-web-app` |

**Hard rules**
- An atom **must not** instantiate another atom or a `*_module`. It calls AWS
  provider resources directly. It may include *tightly-coupled sub-resources*
  that are meaningless on their own (e.g. the S3 bucket atom owns its
  `aws_s3_bucket_versioning`, `_server_side_encryption_configuration`,
  `_public_access_block`).
- A component composes atoms via `module` blocks **only** — never raw `aws_*`
  resources (except trivial glue like a `random_id`).
- A blueprint composes components via `module` blocks **only**.
- Dependencies flow **down by reference**: a higher layer passes IDs/ARNs into a
  lower one. A lower layer never reaches up.

## 2. Secure by default, never stuck

This is the core design principle. Two rules that must both hold:

1. **Every default is the PCI-DSS-compliant value.** A user who sets nothing
   gets a compliant resource.
2. **Every security control is an overridable variable.** Nothing is hardcoded.

To keep insecure choices *intentional and auditable*, an insecure value requires
flipping an explicit, grep-able escape hatch. Controls and their escape hatches
are fields on the single `config` object (see §4); secure defaults are baked into
the `optional()` field defaults:

```hcl
variable "config" {
  type = object({
    enable_encryption = optional(bool, true)  # PCI DSS Req 3
    # ESCAPE HATCH: permit enable_encryption=false. Requires a documented exception.
    allow_unencrypted = optional(bool, false)
  })
  default = {}
}

# in main.tf
lifecycle {
  precondition {
    condition     = var.config.enable_encryption || var.config.allow_unencrypted
    error_message = "Encryption disabled without config.allow_unencrypted=true. File a PCI exception (security@emag.ro) and set the flag."
  }
}
```

Anyone auditing the repo can `grep -r "allow_unencrypted\s*=\s*true"` to find
every place a control was deliberately weakened.

## 3. Directory layout & file layout

Modules live under a cloud-provider root, grouped so the path tells you what a
module is for. Atoms are grouped by **AWS service family**:

```
aws/
  atoms/<service-family>/<name>/      e.g. aws/atoms/kms/kms-key, aws/atoms/ecs/ecs-service
  components/<name>/                  e.g. aws/components/audit-logging
  blueprints/<name>/                  e.g. aws/blueprints/pci-3tier-web-app
```

Source paths: a component references an atom as `../../atoms/<family>/<name>`;
a blueprint references a component as `../../components/<name>`.

Every module directory contains:

```
<module>/
  main.tf          # resources (atoms) / module calls (components, blueprints) only
  variables.tf     # the single `config` object (§4)
  outputs.tf       # the single `manifest` object (§4a)
  versions.tf      # required_version + required_providers (pinned with ~>)
  README.md        # generated by terraform-docs (do not hand-edit between markers)
  examples/
    default/       # minimal PCI-compliant usage — must `terraform validate`
  tests/
    defaults.tftest.hcl   # native `terraform test`
```

## 4. Inputs: one `config` object

Every module exposes **exactly one input variable, `config`**, of `object({...})`
type. No loose top-level variables.

- Secure-by-default controls use `optional(<type>, <secure-default>)` so the
  compliant value is the default. Caller passing `config = {}` (or omitting it)
  gets a compliant resource.
- Inputs the caller *must* decide (e.g. `name`, `vpc_id`) are **required**
  fields — declared with a plain type, no `optional()`. When any field is
  required, drop the `default = {}` so callers can't omit `config`.
- Validate with one or more `validation` blocks on the `config` variable,
  referencing `var.config.<field>`. Each block's `error_message` names the field.
- Escape-hatch fields (`allow_*`) live on the same object, `optional(bool, false)`.
- In a `terraform test`, a `validation` failure is asserted via
  `expect_failures = [var.config]`; a `lifecycle precondition` failure via
  `expect_failures = [<resource>]`.

```hcl
variable "config" {
  description = "All inputs for this module. PCI-compliant defaults baked into optional() fields."
  type = object({
    name              = string                 # required — no default
    enable_encryption = optional(bool, true)   # secure default
    allow_unencrypted = optional(bool, false)  # escape hatch
    tags              = optional(map(string), {})
  })
  # no `default` here because `name` is required
}
```

## 4a. Outputs: one `manifest` object

Every module exposes **exactly one output, `manifest`**, an object collecting
what would otherwise be loose outputs (`id`, `arn`, `name`, …). Callers read
`module.x.manifest.arn`.

```hcl
output "manifest" {
  description = "All outputs of this module, collected on a single object."
  value = {
    id  = aws_thing.this.id
    arn = aws_thing.this.arn
  }
}
```

## 5. Tagging (mandatory)

Tags come from **three layers**, so shared values are set once and never copy-pasted:

1. **Global tags → the AWS provider's `default_tags`** (set ONCE at the root /
   blueprint). These apply to *every* resource in *every* module automatically —
   `Environment`, `Project`, `Owner`, `CostCenter`, `Compliance`, `ManagedBy`,
   etc. Nothing to thread through module calls. This is the answer to "I reuse
   the same tags everywhere": configure them in one place. See
   `aws/examples/global-config/`.

   ```hcl
   provider "aws" {
     region = "eu-central-1"
     default_tags { tags = local.global_tags }  # local.global_tags defined once
   }
   ```

2. **Module-identity tag** — each module hardcodes ONLY its own path, so a
   resource is always traceable to the module that made it:

   ```hcl
   # config.tags = optional(map(string), {})
   locals {
     module_tags = { Module = "atoms/kms/kms-key" }   # this module's path — the ONLY hardcoded tag
     tags        = merge(local.module_tags, var.config.tags)
   }
   ```

3. **Per-resource tags → `config.tags`** — caller overrides/additions for this
   instance only.

AWS merges all three (resource-level `tags` win over `default_tags` on key
conflicts). Do **not** re-declare global tags like `Compliance`/`ManagedBy`
inside modules — they belong in `default_tags`.

> For non-tag values reused across many module calls (region, `name_prefix`, a
> shared KMS key ARN, VPC/subnet IDs), blueprints accept a single `globals`
> object on their `config` and fan it out to each component — set once at the top.

## 6. Versions / pinning

- `required_version = ">= 1.9"` (native test + `lifecycle` preconditions).
- Pin providers with `~>` (e.g. `aws = "~> 5.60"`). Never float unpinned.

## 7. Quality gates (must pass before merge)

```bash
terraform fmt -recursive -check
terraform validate            # per module / example
tflint --recursive
checkov -d . --compact        # PCI/CIS policy scan
terraform test                # per module
terraform-docs markdown table --output-file README.md <module>
```

A control intentionally relaxed via an escape hatch should carry a matching
`checkov:skip` comment citing the exception, so scans stay green *and* documented.

## 8. PCI DSS mapping

Each module README has a **PCI DSS Controls** section listing which requirements
its defaults satisfy (e.g. Req 3 encryption at rest, Req 8 access control, Req 10
logging). See `docs/pci-dss-mapping.md` for the master matrix.

## 9. Naming

- Module dirs: kebab-case singular (`security-group`, not `security_groups`).
- Resource local names: `this` for the primary resource of an atom.
- Variables/outputs: snake_case.
